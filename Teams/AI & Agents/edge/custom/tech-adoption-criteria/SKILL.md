# Tech Adoption Criteria

## Introduction
The gate for adopting emerging technology (new platforms, protocols, infrastructure categories): scored on maturity, ecosystem, fit, and regulatory exposure. Below the bar → watch list with a re-check date; above → a pilot spec to proto.

## Purpose
Emerging tech is a bet with real integration and regulatory costs. The gate converts "should we get into X?" from opinion into a scored, recorded decision — with honest rule-0.6 flags until a real scoring formula exists.

## When to Use
- Anyone proposes adopting/building-on an emerging technology.
- A watch-list re-check date arrives.
- scout routes a platform/infrastructure-shaped find here.

## Structure / Protocol
SCORE four axes, each 1–5 with WRITTEN justification per point:
1. **Maturity** — production track record, stability, breaking-change velocity.
2. **Ecosystem** — tooling, community, hiring/knowledge availability, second-source options.
3. **Fit** — against the operator's venture/stack profiles (`<FILL_IN: profile refs>`): does it serve a named goal or gap?
4. **Regulatory exposure** — with the compliance function's input (`<FILL_IN: compliance contact — Governance's sentinel or operator counsel per config>`).
→ BAR (`<FILL_IN: adoption bar — suggested: no axis below 3 AND total ≥ 14 — RUBRIC, reasoning-based, NOT formula-verified; rule 0.6 flag mandatory on every verdict until logical/ is filled>`) → below → WATCH (watchlist-discipline); above → PILOT SPEC (pilot-spec-handoff → proto).

## Instructions
1. Scores are justified in writing per axis — a bare number is not a score. Unknown = the axis scores its floor (fail closed), with the unknown named.
2. The rubric is a rubric: every verdict carries `[reasoning-based, not formula-verified]` until the logical/ layer has a real decision-analysis source (this is rule 0.6 verbatim — a 1–5 table is not a formula).
3. Venture-specific domains (which technologies even matter) come from operator config, never this skill — the catalog's open-banking example becomes ONE possible operator-selected landscape (see landscape-assets).
4. Verdicts are recorded with their scores and re-check dates; adoption above the bar still flows through the normal gates (pilot → proto's cage → verdict → Rail 3) — this gate opens the pilot door, it never installs anything.
5. Conflicts of interest: if edge scored a tech AND wrote its landscape asset, a second reviewer (forge or meta) sanity-checks the scoring — enthusiasm is a known failure mode of scanners.

## Output Format
Scoring memo: four axes with justifications, total, bar comparison, verdict (watch/pilot), rule-0.6 flag, re-check date.

## Principles
- Unknown scores the floor; enthusiasm gets a second reviewer.
- The gate opens pilots, never installs technology.
- Rubric ≠ formula — say so on every verdict.

## Fallback
Compliance input unavailable? The regulatory axis scores its floor and the verdict says why — regulatory blindness is never scored as regulatory safety.

## Boundaries with Other Skills
- watchlist-discipline manages below-bar outcomes; pilot-spec-handoff packages above-bar ones.
- forge gates AI METHODS (measured benchmarks); edge gates TECH (scored criteria) — an AI-model-shaped candidate goes to forge.
- landscape-assets supplies the domain knowledge scoring draws on.
